12:00:00.001 到 12:00:00.100 预约策略可视化

模型来自默认定时策略：12:00:00.001 触发，先查询场地分布，再生成第一单候选，命中候选后按下单链路提交。毫秒数是可解释预算，真实网络耗时以日志为准。

展示风格
触发点12:00:00.001
观察窗口1-100 ms
默认轮询80 ms
下单间隔30 ms

关键参数

current defaults
window_seconds 60s

每个阶段最多运行 60 秒；两小时目标会先给第一小时 60 秒，再给相邻小时 60 秒。

poll_interval 80ms

普通模式每轮失败或未命中后等待 80ms；服务繁忙时等待不超过 100ms。

step_sleep 30ms

下单链路每个接口之间固定等待 30ms，串联 canBook、getOfferInfo、getUseCardInfo、reservationPlace。

timeout 5s

单次 HTTP 请求最多等待 5 秒；超时会计入失败统计，并进入对应重试或跳过路径。

rounds >=750

轮次数取配置下限和窗口换算值的较大者；60s / 80ms 得到 750 轮。

guide_interval 500ms

引导模式的 collector scheduler 每 500ms 尝试启动一次独立 get_places 探测。

guide_max_inflight 4

最多允许 4 个未完成探测；慢请求不阻塞下一 tick，达到上限时跳过本次探测。

平行场地顺序

default pool
  • 默认场地池排除靠墙 4/5/12 后，顺序为 7, 8, 9, 6, 2, 3, 10, 11, 1。
  • 第一单排序两小时目标下，先比较相邻小时可约总数，再比较单侧最大可约数，再比较场地池顺序，最后比较小时。
  • 100 ms 内影响如果多个场地同一毫秒级都可约，实际提交顺序仍由候选排序决定，不由 API 返回数组顺序直接决定。

边界结论

first 100 ms
  • 快路径100 ms 内通常完成查询、排序、第一候选 canBook，并进入 30 ms 固定等待。
  • 慢路径如果 get_places 超过 70 ms，100 ms 内大概率还没进入正式下单链路。
  • 失败分支如果第一候选返回已被占用，100 ms 内可能切换到第二候选；如果返回操作过快，会睡眠 800 ms，直接越过本窗口。

耗时假设

editable model

当前图按普通网络路径绘制：get_places 约 37 ms，解析排序约 8 ms，canBook 约 20 ms，接口间固定等待 30 ms。要校准真实耗时，取最新 logs/booking_smart_v2_*.log 中每条 [HTTP] 的 elapsed 替换对应区块。
