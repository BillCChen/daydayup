# Daydayup Dual-Account Overview Plan

As of: 2026-07-18 14:47 CST

## Goal

Show the primary and secondary booking accounts simultaneously for account-owned state while keeping shared availability as one common view. `陈启轩` is the default primary/operation account and `张明月` is the secondary account.

## Product Contract

- The header selector becomes an explicit operation-account control; it does not hide the other account's status.
- Token/Session state, card balance, and active bookings are always visible for both accounts.
- Every booking/cancellation row carries its owning account. Cancellation and reconciliation must use that owner, not the currently selected operation account.
- Availability remains a single shared section and is refreshed once through the primary account by default.
- Existing exact booking, manual booking, history, scanning, multi-pool, and cancellation behavior remains compatible.

## Candidate Routes

1. **Shared dashboard with two account lanes** — selected. It supports immediate comparison, preserves one public availability surface, and keeps account ownership visible next to destructive actions.
2. **Duplicate the full dashboard once per account** — rejected. Availability and operational forms would be duplicated, creating conflicting controls and unnecessary network traffic.
3. **Retain selector/tabs with richer labels** — rejected. It still hides one account and fails the simultaneous-display requirement.

## Deployment Routes

1. **GitHub fast-forward plus Web-only Compose rebuild** — primary route. It preserves the scan runtime and provides the cleanest commit provenance.
2. **Verified Git bundle transfer** — fallback if ECS GitHub fetch hangs; validate SHA-256 before fast-forwarding.
3. **Copy changed files directly to production** — rejected because it loses Git provenance and makes rollback/diff verification weaker.

The scan service must not be recreated. A full Compose restart is rejected unless the Web-only dependency graph proves insufficient.

## Risks

- A selected-account global can accidentally overwrite the other account's asynchronous response.
- Cancellation can target the wrong Token if account ownership is inferred from the current selector.
- Duplicating availability queries would add upstream load without adding information.
- Partial failure must remain isolated to one account lane instead of blanking both accounts.

## Completion Criteria

- Both accounts' authorization state, balance, and active bookings appear simultaneously.
- `陈启轩` is visibly marked primary and selected by default; `张明月` is visibly marked secondary.
- Availability appears once and identifies its primary-account refresh source.
- Cancellation always posts and reconciles using the booking's owner.
- Existing regressions and new dual-account browser tests pass.
