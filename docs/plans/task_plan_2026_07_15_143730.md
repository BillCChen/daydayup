# Booking Reliability Repair and Deployment Plan

As of: 2026-07-15 14:44 CST

## Objective

Repair the production booking path exposed by the 2026-07-15 run, publish the validated version, and redeploy Alibaba Cloud without changing the user's booking parameters.

## Confirmed constraints

- Preserve the default user workflow; no manual parameter tuning is required.
- Never automatically replay an indeterminate `reservationPlace` request.
- Keep final booking writes serialized.
- Preserve unrelated local working-tree content.
- Back up production state before replacing containers.
- Do not perform a live booking as deployment verification.

## Candidate routes

### Route A: Increase delays only

- Applicable when failures are solely rate-limit related.
- Strength: minimal change.
- Cost: slower booking attempts.
- Failure mode: the current date-filtered reconciliation still reports false absence, and a timed-out request still terminates the run early.
- Decision: rejected because it does not address the confirmed reconciliation defect.

### Route B: Local exact-match reconciliation plus bounded recovery

- Query recent orders without the ineffective server-side date filter.
- Match date, hour, venue, and active order state locally.
- Require three successful, spaced, read-only snapshots before classifying a timed-out request as stably absent.
- Continue with other candidates only after stable absence; any reconciliation transport failure remains indeterminate and stops further writes.
- Raise the post-success cooldown to 1.8 seconds and make the first `too_fast` response immediately increase the adaptive delay.
- Strength: fixes the production false-negative path while preserving duplicate-order protection.
- Cost: a timed-out write spends several seconds in reconciliation before recovery.
- Failure mode: an order becoming visible only after the bounded reconciliation horizon could still create duplicate risk.
- Decision: selected; multiple successful snapshots and no same-candidate replay reduce that risk.

### Route C: Improve reconciliation but always stop after timeout

- Applicable when duplicate prevention is the only priority.
- Strength: most conservative write behavior.
- Cost: repeats today's early-stop failure even when the request is conclusively absent.
- Decision: rejected because it improves observability but not booking conversion.

## Stages and verification points

1. **Baseline and specification** — preserve hidden local changes, run the existing test suite, and encode the new recovery contract in tests.
2. **Implementation** — repair order reconciliation, bounded recovery, adaptive cooldown defaults, and structured log detail.
3. **Local verification and review** — run focused and full tests, inspect diffs independently from the implementation pass, and verify no real external booking occurs.
4. **Publish** — authenticate GitHub CLI, commit only intended files, push `main`, and create the release tag.
5. **Production deployment** — create a checksummed backup, update the Alibaba Cloud checkout, rebuild/recreate the web and scan services, and preserve state.
6. **Production verification** — verify commit/image version, service health, scheduler state, process topology, and read-only unfiltered order-query behavior.
7. **Closeout** — record exact verification evidence, synchronize deployment records, and check for generated zero-byte/cache artifacts.

## Acceptance criteria

- Reconciliation sends no date filter and locally identifies the intended active order.
- Three successful empty snapshots produce a recoverable stable-absence result; any failed snapshot remains indeterminate.
- No timed-out candidate is replayed automatically.
- A first `too_fast` response increases the cooldown above the prior 1.2-second base.
- Successful booking defaults to at least 1.8 seconds before the next final submit.
- Logs identify every reconciliation snapshot and the final recovery decision.
- Full local tests pass.
- GitHub `main`, production checkout, and the deployed code commit are explicitly recorded.
- Production services are healthy after state-preserving redeployment.

## Risks

- Eventual order visibility beyond the reconciliation horizon could make a stable-absence conclusion incorrect.
- Longer read-only reconciliation consumes part of the booking window.
- Git transport to Alibaba Cloud may fail and require the established bundle transfer fallback.

Stages 1 through 3 are complete. Stage 4, publish, is in progress.
