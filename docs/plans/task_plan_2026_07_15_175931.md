# Multi-pool Booking Implementation Plan

As of: 2026-07-16 11:56 CST

## Objective

Implement, independently review, publish, merge, and deploy a two-account coordinator for manual two-hour direct-fast and guided-fast booking while preserving the single-account v3.7 path.

## Confirmed boundaries

- Branch from aligned `main` commit `d09f5c9` into `multi_pool`.
- Multi-pool accepts exactly two enabled accounts and only `duration=2` with direct-fast or guided-fast.
- The accounts own, pay for, query, and cancel their respective confirmed hours.
- Keep a confirmed single hour when the pair is incomplete; never auto-cancel.
- Do not perform a real booking or cancellation during implementation or deployment validation.
- Never store the supplied login password in Git, plans, logs, command lines, or user state.
- Preserve the three existing skip-worktree analytics files byte-for-byte.

## Candidate routes

1. **Two independent booking processes** — rejected because their success, timeout, and duplicate-prevention states are isolated.
2. **Sequential second-account fallback** — safe but does not materially improve the two-hour contention path.
3. **Single coordinator with two account contexts** — selected because it permits adjacent-hour overlap while centralizing leases, timeout tombstones, and target completion.

## Phases

1. **Baseline and safety artifacts** — complete.
2. **Credential hardening and Web/API contract** — complete.
3. **Engine account contexts and global coordinator** — complete.
4. **Regression tests, security scan, and independent review** — complete.
5. **Commit and push `multi_pool`** — complete.
6. **Merge to `main`, tag v3.8, and publish** — complete.
7. **Alibaba Cloud backup and dry-run-mode deployment** — complete.
8. **Second-account onboarding gate, read-only preflight, live enablement, and closeout** — in progress; investigating the hidden Web onboarding flow and a safe local fallback.

## Account-onboarding routes

1. **Existing Web user-management and OAuth flow** — selected after repairing the panel visibility defect; the password remains transient and only the returned token is saved.
2. **Local one-shot OAuth exchange using the same application methods** — retained as a fallback if the Web exchange fails; obtain the OAuth code interactively, exchange locally without logging credentials, then securely upsert only token/session/card preference to production.
3. **Browser/session extraction from an authenticated mobile profile** — last resort because it is more fragile and exposes a wider browser-session boundary than the application OAuth flow.

The current main route is route 1 because it avoids transferring the new password through chat or shell. Live mode remains unchanged until the acquired account passes read-only card and order preflight.

## Verification points

- Existing 87-test baseline stays green.
- A logical hour has at most one final-submit lease across accounts.
- Only adjacent hours may be in flight; each is owned by a different account slot.
- Timeout reconciliation always uses the submitting account and tombstones the hour across the pool.
- Single-account API and runtime behavior remain compatible.
- Logs and history expose no username, password, token, session, or card value.
- Local, GitHub, and production `main` resolve to the same final commit after deployment.

## Risks

- Upstream throttling may be IP- or service-wide rather than account-specific.
- A delayed order may appear after the reconciliation horizon; timeout tombstones prevent a cross-account duplicate at the cost of giving up that hour.
- Production cannot complete the two-account preflight until the user rotates the exposed password and adds the second account through the existing OAuth flow.
- Direct Git transport to Alibaba Cloud may require the established Git-bundle fallback.

## Convergence decision

Further implementation work cannot safely complete live enablement without the operator rotating the disclosed password and onboarding the intended second account through Web OAuth. The deployed dry-run path, rollback backup, single-account compatibility, and simulated coordinator behavior are verified. After onboarding, the highest-value next evidence is a supervised two-account read-only preflight followed by the first real 12:00 combination log; no synthetic reservation should be created before then.
