# Booking reliability optimization plan

As of: 2026-07-14 20:24 CST

## Goal

Improve the default two-hour direct-fast booking path based on the 2026-07-14 production run, without requiring the operator to tune parameters manually. Preserve first-hour speed while making final-submit throttling, timeout handling, deadline behavior, and diagnostics safer.

## Constraints

- Default parameters must be production-ready; acceptance assumes no manual parameter changes.
- A timed-out `reservationPlace` request has an unknown outcome and must not be blindly resubmitted.
- Preserve unrelated local changes and existing deployment state.
- Do not expose credentials or tokens in logs, tests, or documents.
- Local verification precedes any commit or cloud deployment.
- Production verification must use read-only probes; no live booking attempt is manufactured solely for testing.
- No independent agent review is available in this run. Substitute a separate post-implementation review checklist and targeted regression tests.

## Candidate routes

### Route A: Tune fixed delays only

- Conditions: upstream throttling is stable and transport timeouts are harmless.
- Strength: smallest code change.
- Cost: fewer attempts if the delay is raised globally.
- Failure mode: cannot safely resolve ambiguous POST timeouts and may regress first-hour speed.
- Decision: rejected as incomplete.

### Route B: Adaptive final-submit gate plus unknown-outcome reconciliation

- Conditions: order history can be queried after an ambiguous final-submit result.
- Strength: preserves parallel candidate discovery, adapts only when throttled, and avoids blind duplicate POST retries.
- Cost: requires explicit outcome classification, deadline-aware HTTP behavior, and additional tests/logging.
- Failure mode: reconciliation may be unavailable or delayed, so the result must remain explicit `unknown` rather than guessed.
- Decision: selected main route.

### Route C: Fully serialize the whole booking pipeline

- Conditions: safety is valued above opening-second coverage.
- Strength: simplest rate-limit behavior.
- Cost: sacrifices the concurrency that produced today's first-hour win.
- Failure mode: loses scarce adjacent-hour opportunities before they reach final submit.
- Decision: rejected because it attacks the working discovery path instead of the actual bottleneck.

## Acceptance criteria

- Using defaults, adjacent candidate discovery is not delayed to compensate for final-submit throttling.
- `reservationPlace` transport failures are never automatically replayed by the generic HTTP client.
- A final-submit timeout is logged as an unknown outcome and reconciled against authoritative order data when time permits.
- A `too_fast` response increases only the final-submit cooldown within defined bounds and logs the effective value.
- The scheduler does not begin work that cannot fit within the remaining deadline under its defined safety budget.
- Structured logs distinguish taken, throttled, server failure, transport timeout, unknown outcome, reconciliation result, and deadline exhaustion.
- Existing successful one-hour behavior and two-hour adjacency rules remain covered by tests.

## Major risks

- The upstream order query may have timestamp skew or eventual consistency.
- A request can commit server-side after the client deadline; cancellation is not proof of failure.
- Over-conservative cooldowns can reduce attempts during the opening window.
- Production behavior cannot be fully validated without a real release-time event; deployment verification is therefore structural and read-only.

## Selected behavior specification

- Keep adjacent candidate start at `0s` and discovery concurrency at three.
- Raise the default final-submit throttle base from `0.8s` to `1.2s` without changing the ordinary failed-submit gap.
- Apply at least the throttle base after a confirmed success; increase consecutive `too_fast` gaps by a factor of 1.5, capped at 3.0 seconds.
- Limit each `reservationPlace` send to 2.5 seconds and require at least 0.75 seconds of remaining scheduler budget before sending.
- Disable transport replay only for `reservationPlace`; reconnect/replay remains available to prerequisite calls.
- On final-submit transport failure, wait 0.25 seconds and perform one read-only order reconciliation with a 1.5-second timeout, bounded by the scheduler deadline.
- Match reconciliation by active status, target date, exact start/end time, and court number/name.
- A confirmed match becomes success. An absent or failed reconciliation becomes `unknown_outcome`, is not requeued, and reserves that goal hour against unsafe duplicate final submissions.
