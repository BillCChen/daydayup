# Daydayup Second-Account OAuth Update Plan

As of: 2026-07-18 14:06 CST

## Goal

Generate the existing one-time WeChat OAuth link locally, accept the second WeChat redirect URL from the user, exchange it without persisting credentials locally, verify that it resolves to the intended second booking account, and update only that user's production record.

After the OAuth update, verify the production frontend's complete second-account single-slot workflow by booking one available hour tomorrow, confirming the order under the second account, cancelling it through the frontend, and confirming the resulting refund/state restoration.

Repair the two defects exposed by that live validation: silent/no-effect exact-submit clicks, and cancellation false negatives when the upstream reports failure after applying the side effect.

## Security Boundaries

- Never place passwords, Token values, Session values, card numbers, or redirect authorization codes in Git, plan files, command-line arguments, or logs.
- Treat the redirect URL as a short-lived credential and use it only once.
- Do not modify the primary user's record.
- Before writing production state, create a checksum-verified backup of the current user store and validate the target key.
- After writing, verify identity, enabled state, balance/card lookup, and active bookings using redacted output only.
- Do not restart the continuously running scan process unless the current implementation proves that it cannot reload the user store dynamically.

## Candidate Routes

1. **Existing local Web OAuth flow plus production user-store update** — selected. It reuses the product's tested link and exchange logic while keeping the user in control of the second WeChat authorization.
2. **Complete the flow directly in the production Web UI** — valid but rejected for this operation because the user asked Codex to perform the final write and it gives weaker shell-level backup/verification evidence.
3. **Direct password-based Token acquisition** — rejected because it expands password handling and persistence risk beyond the requested WeChat OAuth flow.

### Live booking/cancellation route comparison

1. **Production frontend exact/single-slot flow** — selected. It exercises the user-visible availability, account selection, order confirmation, active-booking display, cancellation dialog, and refund path requested by the user.
2. **Production frontend manual duration-1 flow** — rejected because it tests the general booking engine rather than the requested single-point interaction.
3. **Direct backend booking/cancellation calls** — rejected because the user explicitly requires frontend interaction and direct calls would bypass the behavior under test.

### Repair route comparison

1. **Backend outcome reconciliation plus frontend submission locks** — selected. Preserve the original account/order, reconcile any non-success/timeout before reporting failure, and make both destructive and booking submissions visibly pending and non-repeatable.
2. **Frontend-only refresh after errors** — rejected because other API clients would still receive false HTTP failures and could retry a cancellation that already succeeded.
3. **Treat every upstream `fail` as cancellation success** — rejected because genuine failures exist; success must be proven by the original account's order state.

| One-shot script could not import project modules when launched from `/tmp` | 1 | Add the explicit current workspace to `sys.path`, re-run AST checks, then relaunch; do not repeat the unchanged invocation. |
| Combined direct order-count query received an unexpected upstream response shape on the primary account | 1 | Keep the already successful second-account verification and use the production Web API as an independent account-scoping check instead of repeating the direct loop. |
| Plan update patch used a stale `As of` value | 1 | Re-read the three plan files and apply the completion update against their exact current contents. |
| Production API probe assumed cookie authentication and received 401 for both users | 1 | Source review shows the UI stores the access key locally and sends `X-Daydayup-Key`; retry once with that exact header, without printing the key. |
| Primary account bookings API returned HTTP 400 while the second account returned 200 | 1 | Record as a separate pre-existing/upstream-shape issue; do not alter the unchanged primary credential during this second-account Token task. |
| The isolated production browser opened at the access-password screen with no existing login state | 1 | Hand the visible page to the user for local password entry; do not read or transmit the password through automation. |
| Browser host telemetry emitted an unrelated Statsig network-size warning after the page snapshot | 1 | The Daydayup page loaded correctly and the snapshot completed; treat the host telemetry warning as non-page noise and continue from the authoritative DOM state. |
