# Daydayup Dual-Account Overview Findings

As of: 2026-07-18 14:57 CST

## Confirmed Requirements

- Account-owned information must be visible simultaneously.
- Shared availability must not be duplicated.
- `陈启轩` is the primary/default operation account.
- `张明月` is the secondary account.
- Previous uncommitted booking/cancellation hardening must be preserved.

## Design Direction

- Register: product UI.
- Scene: a trusted operator checks two booking identities quickly on a phone or desktop under time pressure.
- Use two asymmetric account lanes inside the existing Sage wallet/active-booking section, stacked on mobile and comparable side by side on desktop.
- Keep the existing selector only for future booking ownership and label it as the operation account.
- No new illustrations, modal surfaces, or duplicated functional sections are needed.

## Current Implementation Findings

- The frontend keeps only one mutable `state.primaryCard`, `state.cards`, and `state.bookings`. Each asynchronous loader discards a response when its user key no longer matches `state.selectedUserKey`, so simultaneous account display requires a keyed snapshot store rather than reusing the selector switch path.
- The five-second aligned refresh currently runs status/cards/bookings only for the selected user, while history, scan, job, and availability also share that selector context.
- Active cancellation controls currently store only `bill_num`; the just-completed hardening captures a user key at dialog open, so dual rendering must pass the booking owner's key into `openCancelDialog`.
- The current wallet grid is a single balance block beside one active-booking list. Repeating the entire section would violate the common-availability requirement and the product's anti-density guidance; two account lanes inside this Sage section are the smallest useful structure.
- Desktop currently uses a four-column wallet grid and mobile collapses it. The new account lane container should use a two-column comparison at desktop and a single stacked column below 900px.
- `/api/users` exposes only `default_user_key`; `UserStore.get_user("")` chooses CSV order. A backend role contract is needed so `chen_qixuan` remains primary even if row order changes, while the existing field stays compatible.
- `showAccountDataLoading()` currently clears every selected-account alias and availability when the selector changes. With simultaneous snapshots, switching the operation account should update only selected aliases/history/scan/availability while leaving both overview lanes visible.
- The active-booking expansion flag is global. Dual lanes need per-account expansion state so expanding one account cannot reshape the other.
- The existing Node harness already simulates stale async account responses and cancellation ownership. It is the correct place to add a two-snapshot render contract and a cancellation-owner assertion.
- Existing Python tests have no durable primary-account preference assertion. A temporary CSV ordered with the secondary row first can prove the backend still defaults to `chen_qixuan`.
- `serialize_user()` already includes credential presence and is safe for the dual lanes; no new credential fields or persistent schema changes are necessary.
- Selected-account aliases are consumed by exact-booking balance checks, view-mode details, cancellation fallback, and post-job refresh. The safest route is to keep those aliases synchronized from the keyed snapshot whenever the operation account changes, rather than removing them in the first version.
- Availability selection pricing must continue to use the selected operation account's primary card, even though the displayed availability itself is common.
- The old static frontend test assumes every loader has no parameters and drops non-selected responses. It must be replaced with a stronger contract: explicit account keys populate independent snapshots, while only the selected account updates compatibility aliases.
- `UserStore.get_user("")` returns the first enabled CSV row, so a row reorder can silently move the default away from 陈启轩.
- The safe backend contract can remain schema-free: derive primary/secondary roles from stable user keys and include both keys alongside the compatibility `default_user_key`.
- The live refresh can replace three selector-only operations with one dual-account overview operation while retaining history, scan tasks, jobs, and exact-booking pricing as operation-account scoped.
- Availability currently rejects results after an operation-account switch through `isCurrentUserRequest`; querying it through the stable primary key removes that false invalidation and matches its shared-data meaning.
- Cancellation rendering must look up a booking inside the stored owner snapshot, not `state.bookings`, because the selected-operation compatibility alias may point at the other account.
- The existing wallet DOM should become one full-width account-overview grid rather than retaining separate balance and booking columns; each lane can then contain credentials, balance, and owned bookings together.
- Existing mobile breakpoints already converge below 900px, so account lanes can use a local two-column-to-one-column rule without changing the rest of the workspace.
- The existing `secondaryUserKey` is also the mutable multi-pool participant selection. Role metadata therefore needs a separate `secondaryAccountUserKey`, with a fallback to `secondaryUserKey` only for compatibility tests and pre-role API responses.

## Sensitive Data Rule

Only user keys, display labels, role labels, masked credential presence, balances, and redacted booking summaries may enter tests or durable plans. Passwords, Token values, Sessions, card numbers, OAuth codes, and stable production order identifiers must not be recorded.
