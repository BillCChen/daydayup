## July 19 First Live Multi-Pool Run

- Multi-pool executed correctly as two account contexts: Chen owned 18:00 and Zhang owned 19:00, with Zhang later owning the 17:00 fallback around Chen's confirmed 18:00.
- Chen confirmed 18:00–19:00 on `ymq7`. Zhang's 19:00 path found only one candidate that reached `reservationPlace`; `ymq10` returned explicit `data_error`, while the remaining courts were already taken at `canBook`.
- Zhang's 17:00 fallback then received two `too_fast` responses. A third final submission timed out, and its immediate original-account order query also timed out, so the engine safely stopped with hour 17 unknown.
- A later original-account read shows no Zhang order for the target day, resolving the historic unknown as not booked without any resubmission.
- Authentication is not the failure cause: Zhang completed prewarm, `canBook`, offer, and card calls successfully. Across the run, pool 2 recorded four successful `canBook` calls, four successful offer/card chains, one `data_error`, two `too_fast` final submissions, and one timeout.
- The fixed 350ms pool-2 startup delay consumed competitive time even though the observed `too_fast` responses arose only after pool 2 had already attempted its own final submission. Separate accounts should start together; per-account/global gates remain responsible for subsequent writes.
- Failed-hour history is misleading: the UI shows `ymq3` because `_run_multi_pool_account` logs its final traversed candidate, but the actual 19:00 `reservationPlace` failure occurred on `ymq10`.
- The current immediate timeout reconciliation is deadline-bound; when both the write and first order query time out near the deadline, the job exits unknown and never reclassifies history after upstream recovery.
- Source confirms the failed-court attribution mechanism: a non-terminal `reservationPlace` business failure releases the coordinator hour back to `available` and clears its candidate; the account loop later emits `candidate_exhausted` using the final traversed candidate, even if that candidate never reached the write endpoint.
- The per-account reservation gate currently assigns generic final-submit business failures only the base `350ms` gap, while success/unknown use `1.8s` and `too_fast` escalates from `1.8s`. The live `data_error` therefore did not receive final-write pacing protection.
- Existing timeout reconciliation already carries the original account context and refuses blind replay. The smallest safe extension is a post-loop read-only phase that operates only on coordinator records already marked unknown.
- Web history is event-derived. A resolved hour must emit an updated `multi_pool_slot_result` before the final `multi_pool_complete`; the parser must prefer the last event for a given hour rather than preserve a stale unknown row.
- The coordinator already retains the original owner and candidate for terminal unknown records. A narrow `resolve_unknown` transition can safely move only `unknown → confirmed/tombstoned` and reject all other transitions.
- To report explicit write failures accurately without changing every booking return type, coordinator state can retain `last_attempt_account_slot/candidate/result` even when a failed lease returns the hour to `available`; `canBook`-only misses never overwrite this record.
- Post-run reconciliation should run one bounded read-only worker per unknown hour so two different accounts do not serialize each other's queries. It must continue after an individual query failure, require repeated successful absence before tombstoning, and emit no `reservationPlace` call.
- History parsing currently appends every slot-result event. It must deduplicate by target date/hour and keep the latest event so a delayed confirmed/tombstoned resolution replaces the initial unknown row.
