# Booking Reliability Findings

As of: 2026-07-15 14:44 CST

## Verified production evidence

- The 2026-07-15 run booked 20:00–21:00 successfully but did not book 19:00–20:00.
- After the 20:00 success, the next 19:00 final submit received `too_fast`; the following retry also received `too_fast` because the first adaptive penalty did not increase the 1.2-second base cooldown.
- A later 19:00 final submit timed out after 2.5 seconds.
- Reconciliation used a same-day server filter and returned no orders, after which the scheduler stopped with 19:00 indeterminate despite roughly 20 seconds remaining.
- An unfiltered read-only order query returned the active 20:00 order, proving the filtered query is not a safe source of truth for reconciliation.

## Implementation contract

- Treat unfiltered recent orders plus strict local matching as the reconciliation source of truth.
- Use multiple successful read-only snapshots to distinguish stable absence from an unknown query outcome.
- A stable-absence recovery may continue to other candidates for the same hour but must not replay the timed-out candidate.
- Any failed or incomplete reconciliation remains indeterminate and stops subsequent writes.
- Default cooldowns are code-level safety defaults and do not require user parameter changes.

Their SHA-1 values were unchanged after implementation and full testing.

## Implemented behavior

- Engine version advanced from 3.6.0 to 3.7.0.
- Final-submit success uses a fixed 1.8-second safety cooldown even when the Web payload retains the 1.2-second adaptive base.
- The first `too_fast` response now uses 1.8 seconds, the second 2.7 seconds, and later responses cap at 3.0 seconds.
- Reconciliation removed the ineffective `startTime` and `endTime` query parameters.
- Reconciliation now uses three read-only snapshots at 0.25, 1.0, and 2.5 seconds and logs each decision point.
- Three successful empty snapshots return `reservation_not_confirmed`, which is non-transient and therefore not requeued; other candidates remain eligible.
- A query failure or deadline expiry returns `unknown_outcome`, preserving the existing stop-to-avoid-duplicates behavior.
