## Repair Scope

- The user has now explicitly requested implementation of both observed defects.
- The product contract requires immediate loading/success/failure feedback, deliberate cancellation, and refreshed balance without interpreting raw logs. The repair therefore treats state clarity and duplicate prevention as functional requirements, not visual polish.
- Production deployment is not part of this request. Changes will be implemented and verified locally, leaving the live checkout and containers untouched.
- The selected repair route is backend outcome reconciliation plus frontend submission locks. A frontend-only refresh would leave other clients vulnerable to false failures; blindly accepting every upstream `fail` would hide genuine cancellation failures.
- Exact-submit availability locking is internally inconsistent: `setAvailabilityBusy(true)` disables the button, but `renderExactSelection()` ignores `availabilityLoading` and can re-enable it while the refresh is still in flight. The control also keeps the normal `提交预约` label instead of explaining that availability is refreshing.
- `submitExactBooking()` silently returns when loading or selection state is empty. This matches the live no-feedback symptom and provides no user-facing distinction between duplicate submission and a refresh-driven selection loss.
- The aligned live refresh runs every 5 seconds and treats availability as due every 60 seconds. That makes an availability refresh boundary a reproducible trigger for the first-click symptom: the section becomes inert, while a later selection render can overwrite the submit-button lock.
- The existing JavaScript account-state harness already evaluates the exact-booking functions without binding production event listeners. It can be extended to prove the availability-refresh lock, visible guard feedback, and one-request cancellation invariant without adding a browser dependency.
- The exact payload is built synchronously, but post-result refreshes use mutable global account state. The implementation should capture the original account and selected slots and prevent account/selection controls from changing until the result refresh completes.
- Current cancellation tests cover only an upstream `success` response. They do not cover `fail` with an applied side effect, delayed visibility, unchanged orders, query failure, request timeout, or the one-write invariant.
- Prior UI work established a browser-computed interaction contract and 44px touch targets; this repair should extend behavior checks rather than alter the existing visual system.
