## Initial code observations

- The generic keep-alive client retries once after `OSError` or `HTTPException` regardless of HTTP method, including a non-idempotent final-submit POST.
- Final-submit status `0` is classified as a generic business failure because only HTTP status 500+ maps to server retry.
- Direct waves join every worker thread without a deadline-aware join, so in-flight requests may extend wall time past the configured window.
- The fixed fast-retry delay is 0.8 seconds; today's repeated `too_fast` responses show that value was insufficient under current upstream throttling.
- The engine HTTP timeout default is 5 seconds. Today's three approximately 10-second `reservationPlace` exceptions line up with two back-to-back 5-second sends by the generic retry path. This is verified code behavior plus a high-confidence timing inference.
- The final 8.557-second taken response is consistent with a first 5-second send followed by a second send that returned taken; the exact split is not logged, so this remains an inference.
- The `timeout` argument accepted by `KeepAliveClient.request` is currently ignored; connection creation always uses the client-wide timeout.
- The gate currently applies 0.35 seconds after a success and 0.8 seconds after `too_fast`; there is no adaptive streak or post-success cooldown distinction.
- A started wave blocks on unconditional `thread.join()`. Eliminating the double-send and using a bounded final-submit timeout is therefore more reliable than attempting to abandon live threads.

## Refined main route

- Preserve the 0-second default adjacent candidate start and three-client discovery wave.
- Never transport-retry `reservationPlace`; keep retry behavior for safe prerequisite reads/calls unless tests show otherwise.
- Give `reservationPlace` its own bounded timeout and record a transport timeout as `unknown_outcome`, not a business failure.
- Reconcile an unknown outcome with a read-only `getPlaceOrder` query. If no exact slot match is visible, do not blindly retry that candidate.
- Use the fast-retry gap after a confirmed success, and increase the gap for consecutive `too_fast` responses within a bounded adaptive range.

## Stage 2 acceptance-test map

1. Transport retry control: a simulated `reservationPlace` timeout results in exactly one send; an explicitly retryable prerequisite request may still reconnect once.
2. Per-request timeout: a request-specific timeout is applied to both a fresh and an existing HTTP connection.
3. Adaptive gate: confirmed success uses the safer post-success gap; consecutive `too_fast` responses increase the effective gap with a cap and log the streak.
4. Deadline budget: final submission is skipped before POST when the remaining window is below the minimum safe request budget.
5. Reconciliation confirmed: a final-submit transport timeout followed by an exact active-order match is returned and recorded as success.
6. Reconciliation unresolved: no order match produces `unknown_outcome`, is not requeued, and blocks unsafe duplicate submits for the same one-hour goal.
7. Scheduler summary: unknown hours and completion reason are represented explicitly in structured logs.
8. Default parity: parser and Web command defaults use zero adjacent delay, adaptive 1.2-second base cooldown, and the bounded final-submit timeout without operator input.
